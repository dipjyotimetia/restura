# Chrome Web Store listing copy

Reusable, version-controlled copy for the **Restura Capture** Web Store
submission. Keep this in sync with `public/manifest.json` — if a permission
changes, update the matching justification here.

Listing images live in [`store-assets/`](./store-assets/) (generated by
`npm run gen:store-assets --workspace @restura/extension`).

---

## Single purpose description

> Restura Capture records the network traffic of one browser tab that the user
> explicitly selects, and turns it into a Restura/OpenCollection collection or a
> HAR file for API development and testing. It captures HTTP, GraphQL, WebSocket,
> SSE, and gRPC-web exchanges — including response bodies — redacts secrets
> locally, and lets the user export the result or send it to the Restura desktop
> app over loopback. That is its only function: it does not modify pages, inject
> content, or track browsing.

## Permission justifications

Manifest declares `["debugger", "sidePanel", "storage", "tabs"]` +
`host_permissions: ["http://127.0.0.1/*"]`.

**debugger**

> Capturing full request AND response bodies across HTTP, GraphQL, WebSocket,
> SSE, and gRPC-web requires the Chrome DevTools Protocol (chrome.debugger);
> MV3's chrome.webRequest cannot read response bodies. The extension attaches the
> debugger to only the single tab the user chooses when they press "Start
> capture," and detaches on stop, tab close, or worker teardown. No other tab is
> touched.

**sidePanel**

> The capture interface lives in the side panel: the live list of captured
> requests, the protocol/URL filters, and the Export OpenCollection / Export HAR
> / Send to Desktop / Clear actions.

**storage**

> Used to persist the in-progress (already-redacted) capture session in
> chrome.storage.session so an MV3 service-worker restart mid-capture does not
> lose data, and to store the user's desktop-app pairing code (port + token) in
> chrome.storage.local. No browsing history or personal data is stored.

**tabs**

> Used to resolve the active tab the user selects for capture (its tab ID and
> window) so the debugger attaches to the correct tab and the side panel opens in
> the right window. The extension does not enumerate or monitor other tabs.

**Host permission — `http://127.0.0.1/*`**

> Loopback only. The optional "Send to Desktop" action POSTs the captured session
> to the Restura desktop app's local bridge at http://127.0.0.1:<port>/ingest
> with a user-supplied pairing token. No remote/internet host is requested —
> traffic capture itself uses chrome.debugger, which needs no host permission.

## Are you using remote code?

> **No.** The entire extension is bundled into the package by Vite (background
> service worker + assets). There is no eval, no new Function, no importScripts,
> no remotely-hosted scripts/modules/Wasm, no custom CSP, and no
> externally_connectable. chrome.debugger reads traffic (not code) and the only
> fetch sends a JSON data payload to 127.0.0.1 (not code).

## Data usage disclosures

- **What is handled:** captured request/response data (URLs, headers, bodies)
  from the user-selected tab, and the desktop pairing token.
- **Where it goes:** stays local to the browser/extension. The only outbound
  transmission is the optional Send to Desktop, which is **loopback only**
  (127.0.0.1) — nothing is sent to any remote/internet server by the extension.
- **Secret redaction** runs before any persist, export, or transmit.
- Certifications: does **not** sell user data, does **not** use it for purposes
  unrelated to the single purpose, does **not** transfer it to determine
  creditworthiness or for lending.

---

> Why so few permissions: capture runs through `chrome.debugger` (no host
> permission needed), cookie context is captured via CDP `Cookie`/`Set-Cookie`
> headers (no `cookies` permission), and the only host permission is loopback for
> the desktop bridge. This keeps the listing out of the broad-host "in-depth
> review" path.
