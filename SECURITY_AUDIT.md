# Restura Desktop (Electron) — Security & Robustness Audit

**Date:** 2026-06-21 · **Commit:** `85441ad` · **Branch:** `worktree-security-audit`
**Scope:** The full desktop attack surface for an app that installs on end-user machines — Electron hardening, IPC boundary, SSRF/network guards, secret handling, filesystem/exec, deep-link/auto-update, supply chain, and long-running robustness/performance.
**Method:** Specialist fan-out (7 agents) with an _adversarial-bypass_ mandate (attempt to break each control, not just confirm it exists), every finding carries file:line evidence, and the top findings were independently re-verified by direct code read + an empirical PoC + running the existing security suites (375 tests, all green).

---

## Verdict

The codebase is **mature and security-conscious** — the "easy" Electron mistakes are all handled correctly (see _Strong controls_). The audit found **one verified high-impact SSRF control defect**, **two SSRF redirect gaps**, **two memory-exhaustion (OOM) paths**, and **one secret-export scope gap**, plus medium/low hardening items. None was a remote-code-execution or a renderer→native escape.

**All verified exploitable security gaps have now been fixed in this session** (HIGH H1–H6 + security MEDIUM M1/M2/M3/M4 + the reachable `ws` DoS CVE), each with a regression test or build-verification, and the **full 3160-test suite passes** with no regressions. The remaining open items are **performance/robustness hardening** (M5–M7) and **defense-in-depth / documented platform residuals** — none is a confirmed exploitable security gap.

### Post-review hardening (a self code-review of this PR caught a bug in the H6 fix)

- **H6 follow-up — camelCase secret names were leaking.** The first-cut redactor matched secret field names with separator-anchored regexes (`(^|[-_])token($|[-_])`), so `accessToken`/`secretKey`/`jwt`/`sessionToken` (the **commonest** casing) slipped through _and_ pagination cursors (`page_token`, `next_token`) were wrongly blanked. Reworked `isSecretFieldName` to a segment-based model (splits camelCase + kebab + snake; strong-secret segments always redact; `token`/`key` redact unless qualified by a pagination/structural word). New unit test `keyvalue-secret-redaction.test.ts` pins both the camelCase catches and the pagination/innocuous exclusions.
- **DiD — post-DNS metadata gate.** `assertResolvedAddressAllowed` now refuses a resolved `169.254.169.254` **unconditionally**, so a `*.localhost`/private-literal carve-out can't expose the metadata service even if a hostname resolves to it.
- **Cleanup** — `isPrivateIPv6Groups` now reuses the shared `embeddedV4FromGroups` helper (removed the duplicated embedded-IPv4 extraction).

Second review pass — remaining noted items all addressed:

- **IPC subframe hardening** — `assertTrustedSender` now also rejects any sender frame that has a parent, so a subframe can't drive IPC even at a `/web/index.html` path (the `file:` suffix check is no longer the sole barrier). Test added.
- **WS cross-host redirect** — documented at the `followRedirects` site that same-host handshake redirects are followed while cross-host ones fail closed via the pinned lookup (mechanism covered by `dns-rebind-pinning.test.ts`).
- **De-dup** — extracted the shared credential-header base set into `shared/protocol/credential-header-names.ts` (used by both the AI-context redactor and the export redactor so they can't drift); extracted `createSizeCapTransform` shared by `decodeBodyStream` and `capBodyStream`.
- **Accuracy** — renamed `MAX_SSE_EVENT_BYTES` → `MAX_SSE_EVENT_CHARS` (it bounds JS string length / UTF-16 code units, not bytes).

### H1 · `allowLocalhost:false` is non-functional for literal private/loopback/metadata IPs _(SSRF — verified empirically)_

- **Where:** `electron/main/security/safe-connect.ts:61-66` passes `allowPrivateLiteralHost: isPrivateAddress(host)` for any literal-IP host; `shared/protocol/url-validation.ts:252` then returns early on that flag **regardless of `allowLocalhost`**.
- **Reachable via:** `electron/main/handlers/ai-lab-handler.ts:91-94` and `ai-handler.ts` → `makePinnedFetcher(resolveBaseUrl(provider, baseUrlOverride), { allowLocalhost: isLocalProvider(provider) })`; `shared/protocol/ai/provider-routes.ts:44-45` honors a renderer-supplied `baseUrlOverride` for **any** provider, including cloud. So a **cloud** provider (`allowLocalhost:false`) + `baseUrlOverride: http://169.254.169.254/` reaches the cloud-metadata endpoint, `http://127.0.0.1/`, or RFC1918 — and the response streams back to the renderer (exfil channel).
- **Verification:** confirmed by code read **and** an empirical PoC (`resolveSafeAddress('http://169.254.169.254/', {allowLocalhost:false})` returns the address instead of throwing). The in-code comment at `ai-handler.ts` claiming "cloud providers can never target localhost … the override can't smuggle an SSRF" is **false**.
- **Threat model (honest):** the base URL is user-controlled provider config, so today this is primarily a _confused-deputy / broken-control_ issue rather than a remote SSRF — it requires influencing AI-Lab provider config. It **escalates to CRITICAL** if AI-Lab configs/datasets ever become importable or shareable, or if any programmatic flow sets provider base URLs.
- **Fix:** make `allowLocalhost:false` also forbid private/loopback **literal** IPs — e.g. gate the cloud-provider path through `validateURL(url, { allowLocalhost:false, allowPrivateIPs:false })` before `resolveSafeAddress`, or make `allowPrivateLiteralHost` follow `allowLocalhost` rather than `isPrivateAddress(host)`. **Caution:** the carve-out intentionally allows user-typed LAN targets for _local_ providers (Ollama on `192.168.x`), so the fix must preserve `allowLocalhost:true` behavior and ship with tests (`ai-lab-handler.test.ts`, `dns-guard.test.ts`).

### H2 · WebSocket follows handshake redirects to internal targets unvalidated _(SSRF)_

- **Where:** `electron/main/handlers/websocket-handler.ts:83` sets `followRedirects: true`; on a 3xx `Location` the `ws` lib re-dials reusing the pinned `lookup`, and `createPinnedLookup` (`safe-connect.ts:107-113`) **passes any unexpected hostname straight through to `dns.lookup`** with no re-validation. With `allowLocalhost:true`, a malicious ws server (e.g. from a shared/imported collection) returning `Location: ws://169.254.169.254/` is connected.
- **Exploitability:** higher than H1 — the attacker is just a server the user was induced to connect to, no config control needed.
- **Fix:** `followRedirects:false`, or re-run `validateURL`/`assertResolvedAddressAllowed` on the redirect target and re-pin; alternatively make `createPinnedLookup` _reject_ unexpected hostnames instead of passing through.

### H3 · MCP transport follows redirects unvalidated _(SSRF)_

- **Where:** `electron/main/handlers/mcp-handler.ts:100,113-120` hands `createPinnedFetch(...)` to the MCP SDK transports **without** `redirect:'manual'`, so undici's default `redirect:'follow'` follows a 3xx, again through the pass-through pinned lookup. MCP speaks HTTP, so the redirect target can be the metadata service.
- **Fix:** wrap MCP's fetch with `redirect:'manual'` and route through the shared redirect-follower (as SSE does), or reject unexpected hostnames in the pinned lookup.

### H4 · HTTP text-response path buffers the whole body before the size cap _(OOM / robustness)_

- **Where:** `electron/main/handlers/http-handler.ts:783` (`text: () => readStreamText(bodyStream)`); the `text.length > MAX_RESPONSE_SIZE` check at `shared/protocol/http-proxy.ts:194-195` runs **after** the full string already exists. The binary branch is capped _during_ read (`readStreamToBytes`, `http-proxy.ts:182`) — the common JSON/text branch is not. A chunked response with no `Content-Length` bypasses the pre-buffer guard; the 30s timeout bounds duration, not size, and localhost/LAN upstreams can stream GBs in time.
- **Fix:** cap the text stream incrementally during read, mirroring the binary branch.

### H5 · SSE/MCP frame parser retains an unbounded buffer with no delimiter _(OOM / robustness)_

- **Where:** `shared/protocol/sse-parser.ts:59-75` — compaction only runs when a `\n\n` delimiter advances the cursor; an upstream that streams bytes with no blank-line delimiter grows `this.buffer` without bound in the **main process** for the life of the connection. No max-buffer cap. Same structural pattern in `ndjson-parser.ts` (renderer-side).
- **Fix:** add a hard `buffer.length` cap; abort the stream with an error past it.

### H6 · Collection export redacts only typed auth fields — headers/query/body leak _(secret exposure)_

- **Where:** `electron/main/storage/collection-manager.ts:434-438` and `src/lib/shared/collection-secret-redaction.ts` redact only `auth` blocks. Secrets typed into request **headers** (`Authorization`, `X-API-Key`, `Cookie`), **query params** (`?token=`), or **request body** (`{"api_key":"…"}`) are written verbatim into shared/committed YAML and Postman/OpenCollection exports. Worse, the pre-export warning under-counts: `countCollectionInlineSecrets` only scans auth, so `ExportSecretsDialog` shows `secretCount: 0` and the user's "redact" choice **silently leaves these secrets in**.
- **Fix:** extend both the redactor and the counter to scan header/param rows by denylisted name (reuse `redaction.ts` denylists) + high-entropy token patterns in `body.raw`. Verified note: typed-auth redaction itself is complete; the safeStorage-unavailable plaintext-key fallback is a _documented, warned_ residual (see below), not new.

## LOW / defense-in-depth

- **`connect-src 'self' https: wss:`** — `wss:` is unnecessary on desktop (WebSocket goes via IPC; the renderer's CSP already forbids direct `ws:`); under XSS it permits direct WebSocket exfil bypassing the main-process SSRF boundary. Drop `wss:`. Keep `https:` (renderer-side OAuth2 token exchange needs it). `main.ts:192`.
- **`will-redirect` not handled** — pair it with the `will-navigate` block. `main.ts:204-228`.
- **`shell.openExternal` validates protocol only, not host** — an XSS'd renderer can `window.open('https://attacker/…')` and silently open it in the user's browser. `window-manager.ts:199-209`.
- **Rate-limit / handle-cap gaps:** `collection:watch` (unbounded chokidar `FSWatcher` accumulation), `secret:store`/`vault:set` (on-disk store bloat), `window:new` (unlimited windows), and the `fs:*`/`vault:*`/`collection:*`/`mock:*` handlers lack the rate-limiter every protocol handler has; `grpc:send-message` is uncapped (`z.unknown()`) and unthrottled. All bounded by the XSS prerequisite — DoS hardening, not breaches.
- **macOS `userData`-under-`$HOME` denylist collision** — over-blocks the app's own userData (robustness _false-DENY_, fail-closed, **not** a security hole); masked by the unit test mocking userData as a `$HOME` sibling. `file-operations.ts:131-156`.
- **AI prompt redaction** misses custom-named secret headers (`X-Company-Auth`) and opaque high-entropy body secrets (denylist/pattern-based); env values are blanked wholesale so the common case is covered. Backend `detectUnredactedSecrets` shares the same patterns. `shared/protocol/ai/redaction.ts`.
- **MCP-server `get_history`** returns request URLs verbatim (query/userinfo not sanitized); consent-gated and the headless loader is fail-closed (empty context), so inert until a user opts a history surface in. `src/features/mcp-server/dispatch.ts`.

## Known / accepted residuals (already documented — not new)

- **DNS-rebind during connect (TTL=0 swap):** the guard is pre-flight only; HTTP/gRPC/SSE pin the IP (rebind-resistant on the initial host), Kafka/MQTT narrow the window but don't fully pin. ADR-0006.
- **Linux AppImage auto-update has no binary signature** (electron-updater limitation); integrity rests on pinned-GitHub-over-TLS + `allowDowngrade:false`. mac/win are signature-verified.
- **Kafka/MQTT auto-discovered brokers** (advertised post-connect) are not re-validated. ADR-0006.
- **safeStorage unavailable → plaintext encryption key on disk** (`encrypted-key.ts:129-144`); loudly warned (`log.warn` + persistent `KeychainStatusBanner` + `keychain:status` IPC), documented credential-store policy. Note: `keychain:rotate` is a no-op that won't re-encrypt once a keyring is later installed — worth surfacing in the banner copy.
- **`$HOME` is a writable root** behind a credential-dir denylist — accepted blast-radius for user-picked paths.
- **`vault.get` returns plaintext to the renderer** — intentional Postman `pm.vault.get` parity; sole consumer is the QuickJS sandbox host adapter, distinct from the `SecretRef` handle store.

## Strong controls — verified, do not regress

- **Window:** `sandbox:true`, `contextIsolation:true`, `nodeIntegration:false`, `webSecurity:true`; `will-navigate` blocks all in prod; `will-attach-webview` denied; `setWindowOpenHandler` denies + scheme-allowlists external opens.
- **Build/packaging:** all 7 fuses correct (`runAsNode:false`, `onlyLoadAppFromAsar:true`, ASAR integrity on, `enableNodeOptions/CliInspect:false`, cookie encryption, ad-hoc-sig reset); mac `hardenedRuntime`+`notarize`; win `verifyUpdateCodeSignature`; minimal justified entitlements; `asarUnpack`/`electronDownload` benign; Electron 42 (recent).
- **IPC:** closed, type-checked preload allowlist (no `ipcRenderer` leak, no arbitrary-channel invoke); single `createValidatedHandler` chokepoint does sender-frame validation → Zod → sanitized errors; size caps on every large payload; per-renderer rate limiting with destroy-time eviction.
- **Secrets:** `SecretRef` has **no resolve channel** — plaintext resolves main-only at wire-signing time (ADR-0007 upheld); MCP agent surface exposes auth _type_ only; Sentry scrubs request/user/vars and is opt-in; request-logger redacts URL query/userinfo; the prior tls-passphrase fix holds.
- **SSRF main paths:** HTTP/gRPC/SSE run `validateURL(allowPrivateIPs:false)` + redirect re-validation + IP pinning; the IP-encoding bypass class (decimal/octal/hex/v4-mapped/trailing-dot/userinfo) is dead against the string policy.
- **Filesystem/exec:** git uses `execFile` + `--` arg terminators + a no-leading-dash ref regex (no argument injection) with a canonical-resolved cwd allowlist; `fs:*` is traversal- and prefix-collision-guarded with a credential-dir denylist; mock server is `127.0.0.1`-only, in-memory, no fs/proxy.
- **QuickJS script sandbox:** real 64 MB memory + 5s/30s wall-clock caps, inert in-VM `process`/`require` stubs, value-marshalled host bridges, `pm.sendRequest` routed through the SSRF-guarded proxy — no escape path found.
- **Deep-link / auto-update:** `restura://` is host-allowlisted + `validateURL`'d on URL params, only sink is `webContents.send` (no live renderer consumer today); update feed pinned to GitHub, `allowDowngrade:false`.
- **Robustness:** the `IPC_MODULES` register/dispose invariant guarantees teardown can't drift from registration; per-renderer `bindRendererCleanup`/`disposeByOwner` and every `stop*Cleanup` free real sockets/timers/listeners; most outbound paths carry timeouts; chokidar re-watch accumulation is handled.
- **Empirical baseline:** 375 existing security/guard tests pass at `85441ad`.

## Remaining open items (none is a confirmed exploitable security gap)

1. **M5/M6/M7 (performance/robustness hardening)** — Kafka consumer backpressure, async/debounced electron-store writes off the main thread, SSE idle-read timeout. These are _degradations over a long session_, not crashes or breaches (the two crash-class OOMs, H4/H5, are fixed). Land each with care; none blocks security sign-off.
2. **LOW / defense-in-depth** — drop `wss:` from `connect-src` (left as a recommendation: a prod-only CSP tightening with a small regression risk that warrants a packaged-build check first — and now that the meta-CSP from M1 also carries `connect-src`, both the header and the meta would need the same edit), host-validate `shell.openExternal`, add rate-limiters to `collection:watch`/`secret:store`/`window:new`/`fs:*`, cap `grpc:send-message`, fix the macOS `userData`-under-`$HOME` denylist false-DENY. All bounded by the renderer-XSS prerequisite (itself defended by sandbox + CSP + the pinned IPC trust).
3. **H6 residual** — body-embedded and URL-query secrets (see note above).

_Applied fixes were kept surgical and each shipped with a regression test or build-verification; cross-cutting shared-guard changes (`url-validation.ts`, `safe-connect.ts`) were verified against the full 3160-test suite.
