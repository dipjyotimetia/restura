# Restura Architecture Improvement Roadmap

**Source:** Architectural review on 2026-05-08.
**Strategy:** Six sequential plans. Plan 1 (foundation) must land first — every later plan inherits its files and would otherwise be implemented twice.

> Each plan ships independently and produces working software. Skip or reorder anything past Plan 1 based on need; do not skip Plan 1.

## Plan 1 — Foundation: shared protocol layer

**Issue addresses:** Worker ↔ Electron handler duplication; three HTTP transports; per-protocol IPC sprawl; SSRF guard duplicated in two files that will drift.

**Goal:** Each network protocol implemented once in `shared/protocol/`, consumed by worker handler + Electron IPC handler as thin adapters. Unify URL validation, header policy, body construction, and response shaping. New protocols slot in by adding one file in shared and two ~30-line adapters.

**Deliverables:**
- `shared/protocol/` directory: `url-validation.ts`, `header-policy.ts`, `body-builder.ts`, `types.ts` (`RequestSpec`, `ResponseSpec`, `Fetcher`), `http-proxy.ts`, `grpc-proxy.ts`, `grpc-status.ts`, `mcp-proxy.ts`.
- Worker handlers reduced to ~30 LOC adapters.
- Electron handlers retain Electron-specific features (PAC, SOCKS, interceptors, mTLS) but route validation/sanitisation/response-shaping through shared core.
- `worker/shared/url-validation.ts` and the inline `isPrivateAddress` in `electron/main/http-handler.ts` are deleted.

**This plan is detailed in `2026-05-08-foundation-shared-protocol.md`.**

## Plan 2 — Multi-tab request store + storage consolidation

**Issue addresses:** `useRequestStore` has one `currentRequest` per protocol — users can't open two HTTP requests at once; storage layer split across localStorage, Dexie, and electron-store with inconsistent boundaries; response/request state of truth is fragmented.

**Goal:** Reshape `useRequestStore` around a tab list; consolidate all stores onto Dexie (web) / electron-store (desktop); kill the localStorage path entirely; persist response per tab so restarts don't lose the open response.

**Key tasks:**
1. Define `RequestTab` shape: `{ id, request, response?, scriptResult?, isDirty }`.
2. Replace `currentRequest`/`httpRequest`/`grpcRequest`/`sseRequest`/`mcpRequest` with `tabs: RequestTab[]` + `activeTabId`.
3. Add tab UI primitives (open, close, rename, drag-reorder, "duplicate request").
4. Per-tab Monaco editor model so editor state survives tab switches.
5. Migration: read existing single-request state on first load and seed it as one open tab.
6. Delete `src/lib/shared/storage.ts` (localStorage adapter); migrate any stores still using it to `dexie-storage.ts`.
7. Add variable substitution helpers in URL/header layer: `{{$randomUUID}}`, `{{$timestamp}}`, `{{$randomEmail}}`, `{{$randomInt}}` — match Postman names.

**Why before security/streaming plans:** Plan 3 changes auth signing point — needs the final store shape so signing reads from the right place. Plan 4 streams to a tab — needs the tab model.

## Plan 3 — Security hardening

**Issue addresses:** Encryption key stored next to ciphertext (TOFU theatre); `dangerousPatterns` regex in scriptExecutor blocks legitimate code without security benefit; soft-validation in `updateRequest` lets invalid requests through; AWS SigV4 signs before the worker mutates the request.

**Key tasks:**
1. Replace `getEncryptionKey()` in `dexie-storage.ts` with Electron `safeStorage`-backed key on desktop; on web, prompt for a session passphrase (stored only in memory) with a clearly labelled "skip encryption" fallback. Update README to be honest about web-vs-desktop guarantees.
2. Delete `dangerousPatterns` regex (`scriptExecutor.ts:588-606`). The QuickJS sandbox is the boundary. Add a code-comment explaining why no source-level filter exists.
3. Make `validateRequestUpdate` (`useRequestStore.ts:185`) hard-fail. Display a non-blocking toast on validation error instead of accepting bad state. Prevents silent type-system violations downstream.
4. Move auth signing (especially AWS SigV4) into the Worker / Electron adapter — sign with the exact bytes about to hit the wire. Renderer passes auth *config*, not pre-signed headers.
5. Add web-tier signed-fetch parity for SigV4 — already library-portable via `@aws-sdk/signature-v4`.
6. Document desktop-only features (mTLS, CA cert, SOCKS, system proxy detection) in `docs/ARCHITECTURE.md` and surface a UI hint when a web user toggles them.

## Plan 4 — Streaming end-to-end + gRPC web streaming + HTTP/2

**Issue addresses:** Worker `await response.text()` collapses NDJSON/SSE/large CSV into one blob and can OOM on >20 MB; gRPC streaming methods don't actually stream over the web client; many modern APIs negotiate HTTP/2 and Restura silently downgrades.

**Key tasks:**
1. Worker: switch `proxy.ts` to stream the upstream `Response.body` directly back to the client; add a `streamingMode` flag on the request that bypasses the `MAX_RESPONSE_SIZE` text-cap and instead applies a per-chunk size budget.
2. Renderer: `streamingRequest.ts` already exists in `src/features/http/lib/` — wire it through the executor for `Accept: text/event-stream` and `application/x-ndjson`, render incrementally in the response viewer.
3. gRPC web streaming: use Connect-Web's streaming primitives for server-streaming, client-streaming, bidi. Worker stays out of the path for streams (Connect-Web speaks directly to upstream when CORS permits) — fall back to Worker-as-tunnel only when CORS blocks.
4. Electron: switch from `http`/`https` to a library that supports h2 (`undici` is the right pick — it ships in Node and supports h2 + h3 prep) for non-streaming requests; add an "ALPN" indicator in the response viewer so users see which version negotiated.
5. Backpressure: response viewer must use windowed rendering (not a single scrollable `<pre>`) once total size exceeds 1 MB.

## Plan 5 — CLI runner + JUnit/HTML reporters

**Issue addresses:** No team adopts an API client without CI integration. Bruno's `bru run` is their wedge; Postman's `newman` is theirs.

**Key tasks:**
1. New `restura-cli` package in a sibling folder (`cli/`) that consumes the file-collection schema directly. No coupling to the renderer.
2. Reuse `shared/protocol/http-proxy.ts` (Plan 1) with a Node `undici` fetcher — the CLI is the third backend, and Plan 1 makes it ~50 lines.
3. `restura run <collection-dir> --env <env-file>` runs every request, executes test scripts via QuickJS, prints results.
4. Reporters: `--reporter junit` writes `junit.xml`; `--reporter html` writes a self-contained HTML page; `--reporter json` for downstream tooling.
5. Exit code 1 on any failed test; respects `--bail` for first-failure exit.
6. CI examples in `docs/`: GitHub Actions, GitLab CI, CircleCI.

---

## Plan 6 — Web interceptor parity + plugin model + docs/mock generation

**Issue addresses:** `interceptor-registry.ts` is Electron-only; web users have no extension point; no documentation generation; no mock server (the other two legs of the API-tooling tripod).

**Key tasks:**
1. Promote `interceptor-registry.ts` to `shared/protocol/interceptor.ts`. Worker invokes interceptors at the spec/response boundaries. Web plugins run in the QuickJS sandbox (already in place from scriptExecutor).
2. Plugin manifest: a YAML file in the collection directory declares hooks (`pre-request`, `post-response`, `transform-collection`) that bind to QuickJS scripts. Plugins are versionable in git.
3. Documentation generation: `restura docs <collection-dir>` writes a static HTML site (request list, examples, schemas inferred from request bodies). Reuses the Monaco-rendered request viewer.
4. Mock server primitive: `restura mock <collection-dir>` starts an HTTP server that returns saved example responses for matching request shapes. Useful in CI where the real upstream is unavailable.

## Out of scope (separate roadmaps)

- **Team workspaces / sync.** A real sync story (CRDT over WebDAV/S3, or Yjs-over-WebRTC) is a multi-quarter effort with its own product/protocol decisions. Plan separately when a customer needs it.
- **HTTP/3.** Wait for Node `undici` h3 to stabilise.
- **Mobile.** Out of scope for the desktop+web architecture.
- **Self-hosted Worker for users behind corporate networks.** Already possible (the Worker token model supports it), but needs a packaging story (`docker run resturaproxy`).

Every numbered task here maps to a numbered weakness in the architectural review on 2026-05-08:

- Plan 1 closes review items #1 (handler duplication), #2 (three HTTP transports), #3 (IPC sprawl), #17 (duplicate `isPrivateAddress`).
- Plan 2 closes #4 (multi-tab), #5 (storage split-brain), #7 (response-state truth), #15 (variable helpers).
- Plan 3 closes #6 (encryption key TOFU), #7 (`dangerousPatterns` regex), #8 (soft-validation), #16 (auth-at-the-wire), #18 (desktop-only feature docs), #20 (web interceptor parity initially noted here, fully landed in Plan 6).
- Plan 4 closes #9 (streaming), #10 (gRPC streaming on web), #11 (HTTP/2).
- Plan 5 closes #12 (CLI runner).
- Plan 6 closes #14 (docs gen, mock server), #20 (web interceptor parity / plugin model).
- Items #13 (collaboration/sync) and #19 (rate limiter — already ok) intentionally deferred to "out of scope".
