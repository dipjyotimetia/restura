## Context

An investigation into **repeated macOS keychain permission prompts** during install concluded the prompts were a code-signing / keychain-ACL symptom тАФ a `restura Safe Storage` keychain item first created by a local unsigned `electron:dev` build, whose ACL doesn't match the notarised release's Developer-ID identity тАФ **not** a code bug.

<Aside>
A correctly signed, notarised, non-translocated app prompts once and then stays silent. `safeStorage` uses a **single** keychain item per app and caches the key in memory per process, so "N stores = N prompts" is a myth.
</Aside>

The investigation nonetheless surfaced four hardening gaps worth closing together:

1. **Blocking, rotation-unaware key derivation** тАФ the sync `safeStorage` API treated a *transient* decrypt failure the same as a lost key, regenerating and silently resetting the encrypted stores.
2. **No session permission policy** (Electron security checklist #5) тАФ decisions fell to Chromium defaults.
3. **Non-uniform IPC sender validation** (checklist #17) тАФ `store:set`/`store:clear`, `window:new`, the window controls, and `notification:isSupported` bypassed `assertTrustedSender`.
4. **Silent-optional macOS signing** тАФ CI shipped an unsigned DMG when the Apple secrets were absent, the one scenario that gives end users unfixable repeat prompts.

## Decision

1. **Async, rotation-aware key lifecycle** тАФ `getOrCreateEncryptedKeyAsync` uses `decryptStringAsync`/`encryptStringAsync`, honours `shouldReEncrypt` to re-wrap after OS key rotation, and **pre-warms** all three store keys once at `app.whenReady()` (before the MCP-mode branch). Sync accessors remain a self-init fallback. The loud-warning self-heal for a lost key and the `0o600` plaintext fallback (no OS keyring) are unchanged.
2. **Deny-by-default permission policy** (`permission-policy.ts`) тАФ grants only `clipboard-sanitized-write`; everything else denied.
3. **Uniform IPC sender validation** тАФ every handler asserts the sender frame.
4. **Fail-closed CI signing** тАФ the macOS release leg fails when `CSC_LINK`/`APPLE_ID` are absent.

## Consequences

**Positive**

- Transient keychain unavailability no longer resets the stores; OS key rotation is handled transparently.
- A rendered-response XSS can't silently obtain a device permission.
- IPC sender validation is consistent across the whole surface.
- An unsigned macOS DMG can no longer ship unnoticed.

**Negative**

- Prewarm opens all three stores at startup (one keychain touch, up front) even when secrets go untouched.
- A future renderer feature needing a new permission must be added to the allow-list explicitly.
- The sync key path is retained as a fallback, so two derivation paths coexist (prod always prewarms first).

## Explicitly NOT changed

- **CSP `connect-src 'self' https: wss:`** тАФ load-bearing: the Electron renderer opens `new WebSocket(url)` directly for header-less connections.
- **DNS-rebind residual** for gRPC-reflection / Kafka / Socket.IO / MCP тАФ pre-flight only by necessity; tracked in [ADR 0006](/architecture/adrs/0006-connection-and-dns-hardening/).
- **macOS app-sandbox** тАФ impractical for a tool that makes arbitrary requests and reads user-selected certs/protos; hardened runtime + notarisation is the right bar.

## Alternatives considered

- **Consolidate the three key files into one master key** тАФ rejected: `safeStorage` caches per process, so it's already one keychain touch; zero prompt reduction.
- **Fail hard (never regenerate) on decrypt failure** тАФ rejected: strands the legitimate lost-key recovery; the loud-warning self-heal is the chosen middle ground.
- **Remove the `resetAdHocDarwinSignature` fuse** тАФ rejected: required for fused arm64 builds (electron-builder signs the real Developer ID over the re-applied ad-hoc signature).
